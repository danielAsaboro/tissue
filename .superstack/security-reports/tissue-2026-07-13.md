# TISSUE security audit

Date: 2026-07-13
Mode: Full audit, daily confidence gate (8/10)
Scope: application code, daemon/API, dashboard, analyst/LLM, dependencies, CI, containers, local skill supply chain, and release posture

## Executive assessment

TISSUE is materially safer after this pass. The live engine now admits neither score nor odds
messages until their TxLINE/Solana validation succeeds; it freshly revalidates persisted source
messages during recovery; it neutralizes score-event pressure fields that are not covered by the
stat proof; public errors are sanitized; the analyst has bounded inputs, tool output, provider
responses, concurrency, and request rate; CI actions and container bases are immutable; and the
production dependency audit is clean.

No critical or high-severity exploitable vulnerability met the 8/10 confidence gate. Three
medium findings remain. The most important is a proof-granularity limitation: `validate_stat`
binds cumulative score/red-card values and period to an anchored batch, but does not independently
bind the live minute/final/void fields to one exact Merkle leaf. Current credentialed evidence,
real holdout results, deployment, and the video are also absent, but those are release blockers,
not security vulnerabilities. Repository visibility remains private by explicit instruction.

## Phase 7 — LLM and AI security

User input is sent to the LLM as a user-role message, never concatenated into the system prompt.
LLM output is not executed, used as a URL/path, or passed to a shell. MCP validates tool names and
arguments with bounded Zod schemas. The tool set has exactly three reads; SQLite opens with
`readOnly: true`; the largest ledger query is capped at 500 rows. Provider responses are capped at
2 MiB and the loop is capped at five rounds. API keys remain server-side. The analyst is isolated
from all decision and execution paths.

The new single-instance rate/concurrency limiter prevents unbounded spend in the supplied one-
container deployment. A future horizontally scaled deployment should replace it with a shared
quota service and authenticated per-principal limits.

## Verified findings

### [MEDIUM] SEC-001: Score proof does not bind every decision field to one exact leaf

**Confidence:** 9/10
**Phase:** 9 — Insecure Design / Data Integrity
**Category:** OWASP A06/A08; STRIDE Tampering
**Location:** `apps/daemon/src/exec/scoreAnchorLive.ts:56`, `apps/daemon/src/runtime/liveDesk.ts:18`

**Description:**
The runtime now proves fixture ID, four cumulative stat values, period, proof structure, and that
the streamed timestamp lies inside the anchored batch range. The TxLINE `validate_stat` shape does
not bind the exact live `minute`, `isFinal`, or `isVoid` values to one leaf. Unproved free-kick/shot
pressure is already neutralized before live admission, eliminating its pricing influence.

**Exploit scenario:**
1. An upstream compromise, publisher defect, or authenticated transport-layer corruption produces
   a score payload with correct proven goals/reds but a manipulated minute or terminal status.
2. The four stat proofs remain valid and the timestamp remains within the batch range.
3. TISSUE admits the payload; the minute can change time-decay pricing, while void status can halt
   the fixture. TLS/JWT make this a conditional attack, but the independent on-chain proof alone
   cannot detect it.

**Evidence:**
`assertScoreProofMatchesMessage` checks the timestamp range and stat key/value/period; the proof
response contains no exact message minute/final/void commitment. `admittedSourceMessage` explicitly
neutralizes the separate unproved pressure fields.

**Remediation:**
Require a TxLINE proof response that commits the exact score event/sequence and clock/status, or
combine `validate_fixture` (anchored start time) with an exact event timestamp/status proof. Until
then, treat minute/status as transport-authenticated metadata, never as independently proven data,
and keep settlement/execution surfaces disabled.

**Priority:** P1 (external interface dependent; fix before real-money use)

### [MEDIUM] SEC-003: No signed, gated, reproducible release path

**Confidence:** 10/10
**Phase:** 4 — CI/CD Pipeline Security
**Category:** OWASP A03/A08
**Location:** `.github/workflows/ci.yml:11`

**Description:**
CI validates source and images. A manual, read-only private workflow now retains OCI artifacts
with BuildKit provenance, SBOM attestations, and SHA-256 checksums, but it deliberately cannot push
or deploy. Cryptographic signing, a protected deployment environment, and an exercised rollback
to a known digest remain absent. No deployment exists today, so this is a release-readiness risk
rather than an active production compromise.

**Exploit scenario:**
1. An operator builds/deploys from a dirty or compromised workstation after CI passed another tree.
2. Judges or operators cannot prove which source/lockfile produced the running image.
3. A malicious or broken image cannot be rapidly rolled back to an attested digest.

**Evidence:**
`.github/workflows/private-release.yml` has `contents: read`, `push: false`, immutable action pins,
SBOM/provenance generation and private artifact retention. It has no registry push, cryptographic
signature, approval environment, deploy job, or exercised rollback.

**Remediation:**
When the user selects a private target, publish the already-defined images by digest to its private
registry, sign with keyless OIDC provenance, require a protected environment for deployment,
smoke-test the digest, and retain/exercise the previous digest rollback. Keep repository and
deployment private until the user explicitly authorizes publication.

**Priority:** P1 before deployment (4–8 hours)

### [MEDIUM] SEC-004: Security events have no durable alerting or retention

**Confidence:** 9/10
**Phase:** 9 — Security Logging and Alerting
**Category:** OWASP A09
**Location:** `apps/daemon/src/runtime/liveDesk.ts:278`, `apps/analyst/src/server.ts:122`

**Description:**
Proof, stream, persistence, and analyst failures emit structured logs without credentials. Both
services now expose bounded Prometheus counters and Compose rotates five 10 MiB JSON log files,
but the supplied deployment has no durable external sink or configured alert receiver.

**Exploit scenario:**
1. An attacker or upstream outage repeatedly causes source-proof failures, stream authorization
   failures, or analyst capacity exhaustion.
2. TISSUE fails closed, preventing unsafe outputs, but only container stderr records the pattern.
3. Without durable collection/alerts, the operator may not notice sustained integrity or
   availability attacks and local logs can disappear on container replacement.

**Evidence:**
Structured events, `/metrics`, and bounded local log retention exist. Compose configures no
scraper, alert receiver, or external retention because no private deployment target is selected.

**Remediation:**
Route the existing structured logs and metrics to the selected private deployment provider, alert
on sustained proof/stream failures, SSE pressure, 429s and provider fallback, redact at the sink,
and define its retention/incident runbook.

**Priority:** P1 before unattended deployment (3–6 hours)

11. Replaced source/`tsx` runtime images with compiled daemon/analyst artifacts, production-only
    daemon dependencies, a dependency-free analyst bundle, and Next standalone output; CI now
    starts and checks the compiled services.
12. Added a manual private-only OCI workflow with immutable action pins, maximum BuildKit
    provenance, SBOM attestations, SHA-256 manifests, 14-day private retention, and no write or
    deploy permission.
13. Added bounded Prometheus metrics for source proof, stream, SSE, analyst outcome/rate/fallback
    events and five-file/10 MiB local JSON log rotation.
14. Replaced per-message full-fixture replay and ledger rewrites with the shared engine's
    incremental session, append-only decision records, verified-prefix recovery, and deterministic
    suffix repair. A 195-message comparison measured 72 ms versus 7,390 ms (102×) with an
    identical head hash.
